# Mana Tray
**Status:** Current (as of 2026-07-23)
**Module:** `client/app/board/html/mana-tray.ts`, `client/app/board/geometry/manaTrayProject.ts`, `client/app/board/view.ts`, `client/app/board/html/prompts.ts`

## Problem Statement

Players need to see floating mana without covering battlefield permanents, while spell and payment prompts need their own mana context near the hand/action layer. These are different surfaces and must not collapse into one overlay.

## Solution

Render in-play mana as a world-anchored DOM tray projected by `projectManaTrays(...)`. Compose it in `view.ts` between the vector canvas and the bitmap permanent layer so it belongs to zone furniture. Spell/payment prompts stay in the hand/prompt layer.

## User Stories

- As a player, I can see each seat’s floating mana near that seat’s zone column.
- As a player, I can distinguish floating mana already in the pool from spell/payment prompt UI.
- As a player zooming or panning, mana trays track the table layout.

## Behavior

- `projectManaTrays(players, viewer, count, camera)` converts world anchors to screen positions.
- `manaTrayView` renders a fixed inset, pointer-events-none DOM layer with chip groups at projected positions.
- Chip labels use mana-font glyphs and numeric count overlays for repeated mana.
- The tray is omitted when there are no projected chips.
- Battlefield in-play mana is layer 2: under resting permanents.
- Spell/payment mana belongs with hand/prompt chrome, above hand cards inside that layer.

## Implementation Decisions

- The tray is DOM so mana-font glyphs and labels stay consistent with other chrome.
- It is composed before `MountBitmapLayer`, not inside `boardOverlays`, to preserve layer order.
- The component scales chip font size by camera zoom.
- Accessibility labels describe each chip, such as `{G}` or `2×{U}`.

## Testing Decisions

- Projection tests cover camera pan/zoom and seat anchoring.
- Scene tests assert `mana-tray` appears only when players have floating mana.
- Layer tests or view composition checks should keep the in-play tray before the bitmap permanent canvas.

## Out of Scope

- Client-side mana planning or auto-tap selection.
- Moving spell/payment trays into layer 2.
- New mana source UI beyond what the engine projects.

## Further Notes

- Payment remains engine-side; client previews `auto_tap` but submits action id, target, X, modes, and cost picks.
