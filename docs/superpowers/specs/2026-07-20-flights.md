# Flights

**Status:** Current (as of 2026-07-23)
**Module:** `client/app/board/motion/flights.ts`, `client/app/board/bitmap/mount.ts`, `client/app/board/bitmap/paint-flights.ts`, `client/app/board/submodel.ts`, `client/app/board/messages.ts`

---

## Problem Statement

Playing, resolving, and moving cards must feel continuous without drawing duplicate cards or repainting the entire board every animation frame. Flights need to hide the resting source, retarget when authoritative game state arrives, settle cleanly, and respect reduced-motion preferences.

## Solution

Flights are Mount-painted bitmap card animations owned by the board. `motion/flights.ts` defines pure spawn, step, retarget, and settle helpers. `bitmap/mount.ts` owns the animation clock for the flight canvas and paints only the flight layer on pose ticks. `submodel.ts` updates board state on game sync, spawn, retarget, and settle.

The full board layer stack lives in [`docs/client-canvas-map.md`](../../client-canvas-map.md). Flights are above hand and stack and above resting permanents.

## User Stories

- As a player, when I play a hand card, it travels from my hand toward the stack or battlefield without snapping.
- As a player, when a spell resolves, the card travels from the stack to its destination or disappears into the appropriate zone.
- As an opponent, I see played cards enter from the player area when there is no local hand ghost.
- As a reduced-motion user, I see state settle immediately without glide animation.

## Behavior

### Spawn

A flight is spawned when local action planning or synced game provenance identifies a card movement that should animate. Spawn records card id, optional source hand card id, print/name, screen-space start pose, screen-space target pose, scale, kind, and phase.

Common origins:

- Local hand drop starts from the hand tile's screen origin.
- Local radial or modal action can seed from the selected card or staged origin.
- Opponent plays start from that player's avatar when no hand ghost exists.
- Stack resolve starts from the stack origin at stack scale.

### Retarget

Flights do not invent zone moves. When a snapshot or delta produces the authoritative destination, the existing flight retargets to the card's current layout position or stack position. Retargeting preserves continuity and prevents a separate entrance animation for the same id.

Land plays and stack moves may rebind a flight from source hand id to permanent id. The board keeps the source hand card hidden while the destination id owns the visible flight.

### Step and reduced motion

`stepFlights` advances screen pose and scale toward the target with the existing easing rules. If `prefers-reduced-motion: reduce` is active, the flight snaps to its target and settles without gliding. Settled flights stay in the sync payload long enough for the board model to clear ownership and hidden ids.

### Settle

A flight settles when it reaches the target pose or reduced motion forces the end state. On settle:

- The flight is removed from the live flying set.
- `hideCardIds` no longer hides the resting destination.
- `handHidden` releases the source hand tile when appropriate.
- The owned flight id set is updated so hand, stack, and bitmap paint can resume normal rendering.

Mixed payloads are valid: one `FlightsSynced` message may retain still-flying entries while dropping settled entries.

### Visibility and ownership

`hideCardIds` suppresses resting battlefield/stack faces for ids currently flying. `handHidden` suppresses source hand tiles. The flight-owned id invariant, exposed in the model as `ownedIds` and conceptually as `flightOwnedIds`, prevents competing HTML entrances and duplicate resting faces while a flight owns a card id.

### Paint layer

`board-flight-layer` is a pointer-events-none canvas above hand and stack. `paintFlightCard` draws each live flight from its screen pose using the shared image cache. Resting permanents remain on `board-bitmap-layer`.

### Mount-local animation

The flight Mount owns `requestAnimationFrame` while live flights exist. Each frame:

1. Step flight poses locally.
2. Paint the flight layer only.
3. Emit `FlightsSynced` only when flying membership changes or all flights settle.

Pose-only frames do not run a full board update, do not rebuild `Canvas.view`, and do not repaint the resting bitmap layer.

### Paint gating

`publishBitmapFrame` compares non-flight frame inputs through a resting paint snapshot. Resting bitmap repaint happens when layout, camera, selection, targets, combat staging, payment preview, actions, or hide sets change. Flight pose ticks repaint only the flight layer. Spawn, retarget, and settle still publish normal model state so ownership and hidden ids stay correct.

## Implementation Decisions

- Keep flight math pure in `motion/flights.ts` and DOM-free.
- Keep the animation clock inside the flight Mount, not the board submodel.
- Keep resting paint gated separately from flight pose paint.
- Keep all zone movement authority in game sync and action results; the Mount only animates authorized poses.
- Keep reduced-motion behavior equivalent to immediate settle.
- Keep `hideCardIds`, `handHidden`, and flight-owned ids as the duplicate-rendering guards.

## Testing Decisions

- Pure flight tests cover spawn, retarget, easing, reduced motion, and settle.
- Bitmap frame tests assert pose-only ticks skip resting paint and repaint only the flight layer.
- Story/update tests assert `FlightsSynced` keeps flying ids hidden, clears settled ids, and handles mixed flying/settled payloads.
- Scene tests assert hand and stack faces are suppressed while the flight owns the id.

## Out of Scope

- Changing flight visual design beyond timing and pose parity.
- WebGL, OffscreenCanvas, workers, Pixi, or dirty-rect systems.
- Pointer-move throttling unrelated to active flights.
- Changing engine zone movement or payment semantics.
