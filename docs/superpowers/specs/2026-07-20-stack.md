# Stack
**Status:** Current (as of 2026-07-23)
**Module:** `client/app/board/html/stack.ts`, `client/app/board/geometry/stackLayout.ts`, `client/app/game/intents.ts`, `client/app/board/motion/flights.ts`

## Problem Statement

The board needs a stack surface that stays readable at different viewport sizes, shows incoming spells and abilities, supports targeting into stack objects, and lets the server pause helpless auto-resolve while the player is inspecting the stack.

## Solution

Render the stack as a right-edge DOM overlay with three presentations: pile, expanded strip, and full grid. Geometry helpers choose the presentation and peek spacing from viewport height and stack count. Hover dwell emits a server command, and canvas flights can hide resting stack faces while they own an object id.

## User Stories

- As a player, I can see the top stack object without losing battlefield space.
- As a player, I can expand a crowded stack into readable cards.
- As a player, I can see Island Blue arrows from stack spells/abilities to their declared targets.
- As a player aiming at a stack object, I can click a highlighted stack face to choose it.
- As a player with priority, hovering the stack prevents helpless dwell timing from racing ahead.
- As a player, stack entrances and flight ownership do not create duplicate faces.

## Behavior

- Pile mode peeks cards from the right edge and shows the top object caption or staged target hint.
- Expanded strip shows object names and card art in one row when space allows.
- Full mode uses a centered multi-row grid for large stacks.
- `stackPeekFor`, `stackPresentation`, `stackStripPeek`, and `stackFullPerRow` calculate layout.
- `StackDwellChanged({ dwelling })` fires on stack hover only when the viewer has priority and the stack is non-empty.
- `SetStackDwell` calls `/api/rpc` through `Game.SetStackDwell`.
- `hideFaceIds` / `board.hideCardIds` remove resting faces owned by a canvas flight.
- A staged target ghost appears in pile mode when arrow aiming needs a stack origin.
- Declared `stack[i].target` values paint Island Blue arrows (`stackTargetArrowShapes`) from presentation-aware face origins (`stackFaceScreenOrigin`) to the battlefield permanent or life-orb avatar. Wire carries at most one target per stack entry; multi-target spells show only that projected target.

## Implementation Decisions

- The stack remains DOM in layer 5, while arrows and flights are painted on their board layers.
- The stack aim origin is shared with targeting so arrows point at the same right-edge pile players see.
- Stack→target arrows use `stackFaceScreenOrigin` for the active presentation so expanded/full faces own their arrow tails.
- While arrow-aiming, legal stack faces set `legal-target` and dispatch `TargetChosen` on click.
- Dwell is server state; the client does not locally decide stack-hold outcomes.
- `cardArt(h, opts)` supplies stack card faces and supports optional sizing style.

## Testing Decisions

- Geometry tests cover pile/strip/full presentation and peek calculations.
- Unit tests cover `stackTargetArrowShapes` for object/player targets and skips when unresolved.
- Scene/outcome tests cover legal stack-face highlight and click → submit while Counterspell-style aiming.
- Scene tests cover pile, expanded, staged hint, hold bar, and hidden flight-owned faces.
- Board update tests assert `StackDwellChanged` emits `SetStackDwell`.

## Out of Scope

- Canvas-painted stack cards.
- Client-side simulation of stack-hold timeout.
- New stack targeting rules beyond legal targets sent by the engine.

## Further Notes

- The stack overlay is below prompts, HUD, and the inspect dock in the board layer stack.
- Stack yield controls live in the priority chrome spec.
