Executive summary

Oracle tags are a useful authoring and prioritization layer, not a runtime dependency. They cluster cards by mechanical theme (Historic, enchantment-matters, tribal, sacrifice) in ways that align with FIDELITY_BACKLOG.md increments #63–#90 — but they do not replace the explicit card DSL. The project's ADR 0002 posture (grow filters from real cards) still applies: import otags as catalog hints and gap detectors, not as executable rules.
0 pool cards have no oracle tags — full Scryfall ID coverage in this pool, but tag precision varies: e.g. only 1 card carries synergy-historic while Sram's historic-adjacent payoff is tagged synergy-aura. Tag coverage is community-driven and thematically inconsistent — never gate engine work on tag presence alone.

Recommended uses

  1. Backlog triage. When landing SpellFilter::Historic (#67), query otag historic to find all pool cards that care — not just the 13 listed in the increment table.
  2. Approximate-card clustering. 259 of 259 approximates cards carry otags; group them by shared tag to spot batch authoring wins (e.g. all Spirit-tribal anthems).
  3. Deck-builder search (ADR 0017 extension). Store top-weight otags on CatalogCard for "enchantment matters" / "tokens" browse filters — orthogonal to subtypes.
  4. CI validation tooling. A card tagged historic-matters without a cast_spell trigger using historic filter → lint warning.
  5. Do not: auto-generate [[abilities]] from otags; import the full 4,495-tag ontology; or treat tag weights as rules priority.
